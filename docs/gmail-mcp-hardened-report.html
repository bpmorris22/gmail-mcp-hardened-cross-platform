Cross-Platform Hardened Gmail and Calendar MCP

A public-facing setup, operations, and security guide for a local MCP server supporting Gmail and read-only Google Calendar on macOS, Linux, and Windows.

1. Overview

This server is a local Model Context Protocol (MCP) process that an MCP client launches over standard input and output. It provides multiple-account Gmail operations and read-only Google Calendar access using OAuth credentials held on the user's own computer.

The project is derived from DiegoMaldonadoRosas/gmail-mcp and consolidates locally tested macOS and Windows variants into one portable distribution. Platform-specific setup scripts are included, while the Python server behavior is shared.

Key decisions

  • Keep the Mac variant's functional gmail_send path, with stronger confirmation controls.
  • Support both shell-based installation on macOS/Linux and PowerShell installation on Windows.
  • Keep Gmail attachment sending and received-attachment download within restricted local folders.
  • Exclude cloud-file upload, Drive link generation, and sharing permissions from the MCP.

Email sending is present. The consolidated server exposes gmail_send and delegates delivery to GmailService.send_message. It does not send on the initial request: the server returns a preview and requires a current confirmation token before execution.

3. Cross-Platform Install

Requirements

  • Python 3.10 or later.
  • Claude Desktop or another stdio-capable MCP client.
  • A Google Cloud OAuth Desktop client with the Gmail API and Google Calendar API enabled.

macOS or Linux

git clone https://github.com/OWNER/REPOSITORY.git
cd REPOSITORY
bash setup.sh

Windows PowerShell

git clone https://github.com/OWNER/REPOSITORY.git
Set-Location REPOSITORY
powershell -ExecutionPolicy Bypass -File .\setup.ps1

The setup script creates a virtual environment, installs dependencies, creates the local credential folders and a template-derived config.json if needed, and prints an MCP client configuration using absolute paths.

4. Google OAuth Setup

Enable APIs and create credentials

  1. Create or select a Google Cloud project. A separate project per mailbox is recommended.
  2. Enable the Gmail API and Google Calendar API.
  3. Configure the OAuth consent screen for the intended users.
  4. Create an OAuth Client ID with application type Desktop app and download its JSON file.
  5. Place each downloaded JSON file in the local credentials/ directory, which is excluded from Git.

Configure accounts

Use placeholder account keys and local secret locations in config.json. Never commit this file or OAuth client-secret JSON.

{
  "accounts": {
    "personal": {
      "email": "your.personal@example.com",
      "description": "Personal Gmail",
      "client_secret": "./credentials/personal_client_secret.json"
    },
    "work": {
      "email": "your.work@example.com",
      "description": "Work Gmail",
      "client_secret": "./credentials/work_client_secret.json"
    }
  },
  "credentials_dir": "./credentials"
}

Authenticate

Run once for each configured account, following the browser consent flow.

# macOS or Linux
.venv/bin/python setup_auth.py

# Windows PowerShell
.\.venv\Scripts\python.exe setup_auth.py

The server requests these scopes only:

https://www.googleapis.com/auth/gmail.modify
https://www.googleapis.com/auth/calendar.calendarlist.readonly
https://www.googleapis.com/auth/calendar.events.readonly
Scope changes require re-authentication. Users who tested an earlier local variant should run setup_auth.py again so new tokens reflect the consolidated scope set. This version requests no Google Drive scope.

5. MCP Client Configuration

The MCP client starts the Python server as a child process and communicates with it over stdio. No standalone web server or open listening port is required.

macOS or Linux example

{
  "mcpServers": {
    "gmail-hardened": {
      "command": "/absolute/path/to/gmail-mcp/.venv/bin/python",
      "args": ["/absolute/path/to/gmail-mcp/server.py"]
    }
  }
}

Windows example

{
  "mcpServers": {
    "gmail-hardened": {
      "command": "C:\\absolute\\path\\to\\gmail-mcp\\.venv\\Scripts\\python.exe",
      "args": ["C:\\absolute\\path\\to\\gmail-mcp\\server.py"]
    }
  }
}

Common Claude Desktop configuration locations:

  • macOS: ~/Library/Application Support/Claude/claude_desktop_config.json
  • Windows: %APPDATA%\Claude\claude_desktop_config.json

6. Security Model

Confirmation-bound writes

All tools that create or modify state use a two-step flow: gmail_send, gmail_create_draft, gmail_modify_labels, gmail_trash, and gmail_download_attachment. The initial call returns a preview and a ten-minute, one-time confirmation_token. Execution requires identical arguments together with confirm: true and that token.

Untrusted external data

Mailbox content and calendar content can contain prompt-injection text. Tool responses containing external text include an untrusted-content warning, instructing the MCP client to treat it as data and never as authorization to send, delete, relabel, download, or disclose material.

Attachment controls

  • Outbound attachments and downloaded attachment destinations are limited to the user's Downloads, Documents, or Desktop folders.
  • Paths are resolved before checking, preventing a symlink from escaping an allowed directory.
  • The configured credentials directory, configured client-secret files, and common private-key or secret filenames cannot be attached.
  • Downloaded bytes remain untrusted and are not automatically opened or executed.

Local credential protection

OAuth client secrets, refresh tokens, and user configuration remain local and are excluded from Git. Token writes request user-only filesystem permissions where supported. On shared Windows computers, users should also review folder ACLs on the project and credentials/ directory.

Residual risk

This MCP reduces unintended writes but cannot defend against theft of local OAuth tokens, a compromised workstation, or a user authorizing a harmful preview. Review the complete threat model in SECURITY.md.

7. Merge Decisions

Source materialCarried forwardChanged or excluded
macOS-oriented hardened guide and implementation Gmail send flow, Calendar access, multi-account configuration, setup pattern Examples sanitized; confirmation controls strengthened
Windows implementation and session report PowerShell setup and cross-platform operational notes Google Drive experiments, sharing links, and rich-composition experiments excluded
Final consolidated version One portable server, platform setup scripts, security documentation and CI One-time mutation confirmations, least-privilege Calendar scopes and dependency audit added
Why no Google Drive? Uploading a file and granting access is a separate disclosure boundary from reading and sending email. It should be implemented in a dedicated MCP with its own OAuth scopes, consent decisions, logging, tests, and security review.

8. Verification and Publication

The consolidated repository is designed to contain source, documentation, setup scripts, and tests only. Local assembly inputs, credential material, tokens, private configuration, virtual environments, generated archives, and historic source reports are excluded from publication.

Validation performed for the initial release

  • Smoke tests passed locally for the registered MCP tool surface.
  • Dependency audit reported no known vulnerabilities in pinned direct dependencies.
  • GitHub Actions validated the project on Windows, macOS, and Ubuntu.
  • The final source review confirmed gmail_send remains available and Google Drive functionality is absent.

Before publishing changes

  1. Check that no config.json, credentials/, token, client-secret, local report, or archive file is staged.
  2. Run python -m unittest discover -s tests.
  3. Run a dependency vulnerability audit and review changes to OAuth scopes or write-tool behavior.
  4. Re-authenticate test accounts after any scope change and perform a deliberate preview-confirm send test.
